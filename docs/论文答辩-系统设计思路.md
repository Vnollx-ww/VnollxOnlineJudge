## 1. 一句话 + 术语小词典

**一句话**：这是一个 **网上刷编程题、自动判对错、办比赛排名、发题解、好友私信、系统通知、多模型 AI 答疑** 的完整平台；用户代码在 **隔离环境** 里跑，高峰用 **排队** 扛住提交潮。

**常用缩写（答辩老师可能问）**：

| 词 | 人话 |
|----|------|
| OJ | Online Judge，在线判题系统 |
| AC / WA / TLE / MLE / RE / CE | 通过 / 答案错 / 超时 / 超内存 / 运行错 / 编译错（中文界面里也可能显示「答案正确」等） |
| JWT | 登录后发的「门禁卡字符串」，带在请求里证明身份 |
| RBAC | 角色（如管理员）绑一堆权限，用户绑角色 |
| REST API | 网页用 HTTP 调后端的固定地址（如 `/api/v1/...`） |
| WebSocket | 浏览器和服务器 **长连接**，适合实时推送（判题结果、通知、聊天、AI 流式字） |
| SSE | 服务器一段段推文字，AI 代理常用 |
| Docker | 把程序和环境打成「标准箱子」，换机器也好跑 |
| 沙箱 / Go-Judge | 专门跑用户代码的隔离环境，限制时间内存 |

## 2. 功能全景：业务模块一个不漏

### 2.1 用户与账号

- 注册、登录、忘记密码（常配合 **邮箱验证码**，见邮件接口）。  
- 个人资料：头像、昵称、签名等；头像等大文件走 **MinIO**。  
- 设置页：改资料、改密码等（以实际页面为准）。  
- 学习数据：提交数、通过数、通过率、已通过题目、个人主页统计图表等。  
- **全站排行榜**（`/ranklist`）：用户间解题排名展示。

### 2.2 题目与标签

- 题库列表：分页、难度、标签筛选、搜索。  
- 题目详情：描述（多支持 Markdown）、样例、时间/内存限制、标签。  
- 标签：独立列表与维护（管理员在后台管标签）。  
- 题目数据：测试点常以 **zip** 等形式存 **MinIO**，判题时按题目 id 取数据包。

### 2.3 评测与提交

- **正式提交**：代码入队异步判题，状态从「等待评测」→「评测中」→ 终态。  
- **测试运行**：样例或自定义输入，可走同步短路径，减轻队列压力。  
- **多语言**：C++、Java、Python（实现上通过不同 **判题策略** 对接 Go-Judge）。  
- **防刷**：同一用户同一题约 **3 秒内** 重复提交会被 Redis 锁拒绝。  
- **提交记录**：列表、筛选、详情；比赛下有单独的提交列表页。

### 2.4 竞赛（Contest）

- 竞赛列表与详情：公开/私有、时间、说明。  
- 报名、赛中做题、赛中提交。  
- **实时排名 / 榜单**：与 **ACM 风格罚时** 相关逻辑；赛时大量用 **Redis** 算排名，减轻数据库压力。  
- 竞赛专用题目页路由：`/competition/:cid/problem/:id`。

### 2.5 练习（Practice）

- 练习集列表、练习详情、练习内题目（与比赛类似但偏「训练包」场景）。  
- 后台可创建/编辑练习、往练习里加题。

### 2.6 题解（Solution / Solve）

- 某题下的题解列表、题解详情（Markdown、代码高亮等）。  
- 发布题解、全站题解入口 `/solutions`。  
- 管理员可 **审核** 题解（权限 `solve:audit`）。

### 2.7 评论

- 挂在题解等内容下的评论（创建、删除需对应权限）。

### 2.8 社交：好友与私信

- 搜用户、发好友请求、同意/拒绝、好友列表。  
- 好友间 **私信**；前端有 `MessageWebSocketProvider` 做实时消息相关上下文。

### 2.9 通知

- 系统通知列表、未读、详情。  
- 管理端可发通知；可通过队列 `notification` 相关能力与 WebSocket 推送配合。

### 2.10 AI 助手

- 多模型选择（后台配置 API Key、模型名、**代理类型** 国内/海外）。  
- 会话列表、历史消息、**流式打字**效果；部分模型支持 **思考过程** 展示。  
- 题目页可 **带入题目上下文**；个人主页可有 **AI 学习建议** 入口（自动发问）。  
- Python 代理里 **OjTools** 可查库回答事实问题（只读、白名单函数）。

### 2.11 管理后台（`/admin/*`）

前台独立布局，子模块与代码文件对应关系：

| 后台页面（前端） | 大致能力 |
|------------------|----------|
| AdminUsers | 用户管理 |
| AdminProblems | 题目管理 |
| AdminCompetitions | 比赛管理 |
| AdminPractices | 练习管理 |
| AdminSolves | 题解管理（含审核等） |
| AdminStatistics | 统计数据 |
| AdminRoles / AdminPermissions | 角色与权限分配 |
| AdminAiModels | AI 模型与平台配置 |

### 2.12 其他前台页面

- **首页** `/`、**关于** `/about`。  
- **CodeArena** `/codearena`：独立入口页面（与主布局分离）。  
- **登录** `/login`、**注册** `/register`、**忘记密码** `/forgot-password`。

## 6. 判题子系统：从提交到出结果

### 6.1 涉及的主要类（包名简写）

- `JudgeController` → `JudgeService` / `JudgeServiceImpl`：接收提交、建 `Submission`、**Redis 防重复提交**、**雪花 ID**、调用 `JudgeProducer`。  
- `JudgeProducer`：把 `JudgeInfo` 序列化发到 RabbitMQ（见下节）。  
- `JudgeConsumer`：监听 **`submissionQueue`**，反序列化 → 改状态「评测中」→ `JudgeStrategyFactory` 选策略 → `judge(...)` → 更新 `Submission` → `processSubmission` → **WebSocket 推送**。  
- `judge/JavaJudgeStrategy`、`CppJudgeStrategy`、`PythonJudgeStrategy`：各语言如何组请求调 **Go-Judge**。  
- `JudgeStrategyFactory`：按语言字符串选实现。  
- `SubmissionServiceImpl.processSubmission`：根据是否 AC、是否首次通过、是否比赛 (`cid!=0`) 更新 **用户统计、题目统计、Redis 竞赛键** 等。

### 6.2 状态流转（口语）

等待评测 → 评测中 → 答案正确 / 答案错误 / 超时 / 超内存 / 编译错误 / 运行错误 …（具体字符串以 `RunResult` 与前端展示为准）。

### 6.3 测试运行

`JudgeController` 的 `test` 接口：`testSubmission` 直接 `testJudge`，可走样例或自定义输入；`customTest` 为 true 时侧重返回 **实际输出** 给用户对拍。

## 7. 消息队列 RabbitMQ：交换机、队列、路由

（依据 `RabbitMQConfig`、`JudgeProducer`）

| 类型 | 名称 | 说明 |
|------|------|------|
| 交换机 | `judge`（Direct） | 判题任务分发 |
| 路由键 | `judge.submit` | 生产者 `send("judge", "judge.submit", message)` |
| 队列 | `submissionQueue` | 消费者 `@RabbitListener(queues = "submissionQueue")`；**支持优先级** `x-max-priority: 10` |
| 交换机 | `notification`（Direct） | 通知类消息 |
| 路由键 | `notification.send` | 绑定 `notificationQueue` |
| 队列 | `notificationQueue` | 通知消费用 |
| 队列 | `replyQueue` | 配置中存在，具体 RPC/回复场景以业务代码为准 |

**人话**：判题就是「厨房订单」从 `judge` 交换机根据 `judge.submit` 路由到 `submissionQueue`，厨师（消费者）一条条做。

## 8. Redis 在本系统里干什么

列举 `SubmissionServiceImpl` 等中出现的 **键模式**（比赛相关为核心）：

- `submission:user:{uid}_{pid}`：分布式锁，**防重复提交**（约 3 秒）。  
- `competition:{cid}:problems`：比赛题目信息 JSON，赛中取题用。  
- `competition_user_pass:{cid}:{userName}`：用户通过题数。  
- `competition_user_penalty:{cid}:{userName}`：罚时相关。  
- `competition_problem_pass:{cid}:{pid}`：该题通过情况。  
- `competition_problem_submit:{cid}:{pid}`：该题提交情况。  
- `competition_ranking:{cid}`：榜单用 **有序集合** 等结构。  
- `competition_time_out:{cid}`、`competition_time_begin:{cid}`：结束/开始时间，用于 **TTL** 与罚时计算。

另有：**权限缓存**、其它业务缓存（以 `RedisService` 使用处为准）。

## 9. 权限 RBAC：权限码全表 + 怎么生效

### 9.1 权限码常量（`PermissionCode.java` 全量）

**用户**：`user:view` `user:create` `user:update` `user:delete` `user:manage`  
**题目**：`problem:view` `problem:create` `problem:update` `problem:delete` `problem:manage`  
**比赛**：`competition:view` `competition:create` `competition:update` `competition:delete` `competition:manage`  
**练习**：`practice:view` `practice:create` `practice:update` `practice:delete` `practice:manage`  
**提交**：`submission:view` `submission:view_all` `submission:rejudge` `submission:submit`  
**题解**：`solve:view` `solve:create` `solve:update` `solve:delete` `solve:audit`  
**标签**：`tag:view` `tag:create` `tag:update` `tag:delete`  
**通知**：`notification:view` `notification:create` `notification:delete`  
**社交**：`friend:use` `comment:create` `comment:delete`  
**AI**：`ai:config_view` `ai:config_update` `ai:chat`  
**系统**：`system:settings` `system:monitor` `system:log`  
**角色权限**：`role:view` `role:create` `role:update` `role:delete` `permission:assign`

### 9.2 怎么生效（人话）

- 接口方法上写 `@RequirePermission(某个常量)`。  
- **`PermissionAspect`**（AOP）在运行前查当前用户是否具备该权限（常配合 **Redis 缓存** 权限结果）。  
- 前端再根据同一套码 **隐藏按钮**，但**真正安全靠后端**。

## 10. 过滤器、AOP、异常：横切逻辑

| 组件 | 文件/包 | 人话 |
|------|---------|------|
| Token 过滤 | `filter/TokenFilter.java` | 解析 JWT，未登录拦不拦由路径规则决定 |
| XSS 过滤 | `filter/XssFilter.java` + `XssHttpServletRequestWrapper` | 清洗危险脚本 |
| 布隆过滤器 | `filter/BloomFilter.java` | 可做存在性预判等（具体用途看注册配置） |
| 权限切面 | `aop/aspect/PermissionAspect.java` | 校验 `@RequirePermission` |
| 限流切面 | `aop/aspect/RateLimitAspect.java` | 接口限流（Redis） |
| 管理员检查 | `aop/aspect/AdminCheckAspect.java` | 管理类操作额外约束 |

## 18. 答辩问答扩展版

**问：队列名为什么和消费注解一致？**  
答：`submissionQueue` 在配置里声明并绑定交换机，消费者监听同一队列名，保证消息不丢单。

**问：优先级队列干什么用？**  
答：`x-max-priority` 预留 VIP 或紧急重判等扩展，当前代码里 `priority` 可先固定传值。

**问：雪花 ID 好处？**  
答：分布式下唯一、同一提交在库/MQ/推送可对齐。

**问：为什么判题不放在 Java 里跑？**  
答：安全隔离 + 专业沙箱；Java 进程与用户代码必须隔离。

**问：权限存在哪里？**  
答：数据库角色权限表；查询结果可缓存 Redis；接口用 AOP 强制校验。

**问：前端为什么还有权限？**  
答：只为体验，**不能当安全**；安全以后端为准。

**问：AI 如何防止乱查库？**  
答：仅 21 个白名单工具 + 只读；带 `current_user_id` 上下文。
