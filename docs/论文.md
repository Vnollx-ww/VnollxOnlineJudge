# 1 概述

## 1.1课题背景及意义

在线编程评测系统（Online Judge, OJ）是一种面向算法与程序设计训练的自动化评判平台，其核心工作流程为：接收用户提交的源代码后自动完成编译、执行，并将程序输出与预期答案逐测试点比对，最终在秒级时间内返回"通过"（Accepted）或"错误"（Wrong Answer）等判定结果。随着 ACM-ICPC 竞赛和互联网企业技术面试的推动，LeetCode、Codeforces、洛谷等在线评测平台的用户规模已达百万级，OJ 系统逐渐成为高校编程教学和算法训练中不可或缺的基础设施。

然而，当前多数高校在编程实训环节仍依赖人工批阅代码的方式，该模式存在两方面不足。其一，反馈延迟：教师难以在课堂时间内逐一审阅数十份乃至上百份代码提交，学生往往在错误产生数天后才获得评判结果，错过了最佳的认知纠偏窗口。其二，缺乏个性化诊断：面对同一道题目，不同学生的错误类型差异显著——有的是算法策略失误，有的是边界条件遗漏——人工批阅难以为每位学生提供针对性的修复建议。上述问题构成了本课题的直接驱动力。

基于此，本系统的设计目标可概括为以下四个维度。第一，自动化评测与即时反馈：构建异步评测管线，利用消息队列调度判题任务并经 WebSocket 实时推送结果，从而将教师从重复性代码审阅中释放出来。第二，算法训练与赛事支撑：提供题库管理、多难度分级、标签检索等功能，同时内置符合 ACM 罚时规则的比赛模块，满足校内选拔赛与专项训练需求。第三，AI 辅助学情诊断：集成多大语言模型（LLM），借助 Function Calling 机制使 AI 能够查询用户的提交轨迹与薄弱标签，进而生成个性化题目推荐与学习路线。第四，安全隔离与并发稳定性：采用内核级沙箱执行用户代码，配合读写分离与缓存策略保障系统在高并发场景下的可用性。

## 1.2 国内外研究现状

在线评测系统的起源可追溯至 20 世纪 60 年代，彼时美国高校已在大型机环境中试验自动化代码评判原型。经过半个多世纪的演进，OJ 系统在架构形态、评测安全性以及智能化程度方面均发生了深刻变化。

**国外研究现状：**

UVa Online Judge 是最早向全球开放的在线评测平台之一，其题库覆盖了大量 ACM-ICPC 经典赛题，至今仍被视为算法竞赛入门的重要训练资源。Codeforces 由俄罗斯团队于 2010 年前后上线，以高频次赛事（平均每周两至三场）和严格的分级积分制度（Elo Rating）著称，注册用户已逾百万；其评测后端基于自研沙箱并支持数十种编程语言，在并发处理与防作弊方面积累了丰富经验。LeetCode 则面向软件工程师的技术面试场景，题目偏重工程实践与数据结构应用，其在线编辑器与即时反馈机制对后续平台的交互设计产生了显著影响。上述三类平台在技术方向上呈现共性特征：多语言编译支持、大规模题库管理、积分排名体系以及实时比赛调度。

**国内研究现状：**

国内 OJ 生态的早期代表为北京大学的 POJ 与浙江大学的 ZOJ，两者为中国 ACM 竞赛选手的培养提供了核心训练环境。近年来，洛谷与牛客网等新一代平台在用户界面与社区协作方面做出了改进——洛谷引入了题解区与讨论区以促进知识共享，牛客网则将 OJ 功能与招聘流程深度整合。在技术架构层面，上述平台普遍开始从单体架构向微服务或容器化部署迁移，以应对弹性扩缩容需求。

综合国内外现状，当前 OJ 系统的演化呈现四个趋势：（1）评测执行环境从进程级隔离向基于 Linux cgroup/namespace 的容器级沙箱演进，安全防护粒度更细；（2）系统架构由单体向微服务或异步消息驱动方向发展，以提升横向扩展能力；（3）大语言模型的兴起推动 AI 辅助功能从简单的代码补全扩展至错误诊断与学习路线规划；（4）前端交互强调低延迟、沉浸式的编码体验，在线 IDE 与实时调试逐渐成为标配。然而，现有商业平台大多为闭源系统，不支持高校私有化部署；而开源方案在 AI 能力集成深度上仍有明显不足——多数仅停留在单模型的代码补全层面，尚未实现基于用户学情数据的智能体（Agent）推理。本系统正是针对这一空白展开设计。

## 1.3 本课题主要工作

本课题的研究工作历时约五个月，涵盖需求调研、方案设计、系统开发与测试验证四个阶段。下面从需求分析出发，逐步阐述各阶段的工作内容。

**需求调研与对比分析：** 项目前期对 LeetCode、洛谷、Codeforces 等主流平台进行了功能梳理，并面向校内学生收集了关于评测延迟、错误信息可读性以及 AI 辅助需求等方面的反馈。在此基础上，将已有系统归纳为两类进行对比：

**传统系统（以 POJ/UVa 为代表）：** 题库资源丰富且经过长期积累，但交互层面存在明显短板——界面设计停留在早期 Web 1.0 风格，评测结果仅返回简短状态码（如 WA、RE），缺乏面向初学者的错误上下文提示；此外，其评测沙箱多采用进程级限制，对 fork 炸弹、文件系统越权等攻击手段的防御能力有限。

**现代商业平台（以 LeetCode/洛谷为代表）：** 在用户体验和功能完整度上有显著提升，但闭源商业模式使得高校难以进行私有化部署和二次开发。尽管部分平台已尝试引入 AI 辅助，但通常仅绑定单一模型、仅覆盖代码补全场景，未能与系统内部的用户学情数据深度耦合。

**本系统在上述对比基础上，着力解决以下技术问题：**

**异步评测与实时反馈机制：** 后端启用 Java 21 虚拟线程（Virtual Threads），将提交请求经 RabbitMQ 的 DirectExchange 路由至持久化的 submissionQueue，由 JudgeConsumer 异步消费。评测完成后通过 JudgeWebSocketHandler 基于 ConcurrentHashMap 维护的用户会话映射向客户端实时推送判题状态，有效消除了同步等待瓶颈。

**内核级安全沙箱：** 评测执行层采用 Go-Judge 沙箱，其底层利用 Linux cgroup 与 namespace 对用户进程实施 CPU 时间、内存用量及进程数的硬限制。以 C++ 评测策略（CppJudgeStrategy）为例，编译与运行分两阶段向 Go-Judge 的 /run 端点发起 HTTP 请求，编译产物以 fileId 形式暂存于沙箱文件系统，运行阶段按批次并行执行测试用例（每批上限 20 组），执行完毕后即时清除编译产物，防止资源泄漏。

**多大语言模型接入与智能体（Agent）架构：** 系统构建了独立的 Python FastAPI 代理服务（ai-proxy），统一封装 Gemini、DeepSeek、智谱 AI、Mistral 等厂商的 API 调用逻辑。Java 后端的 AiController 以 SSE 协议暴露流式聊天接口，请求经 HTTP 转发至 Python 代理后，由 provider_registry 模块按模型名路由至对应厂商的 stream 方法并逐 chunk 返回。

在此基础上，代理层的 /v1/chat/stream/tools 端点实现了 Function Calling 回合：AI 可在对话过程中自主触发预定义的只读工具函数（如 getUserProgress、searchProblems 等共 21 个），对评测数据库执行结构化查询，获取用户在各标签维度的提交与通过统计后生成个性化的题目推荐与学习路线规划方案。工具调用经 execute_tool 白名单机制执行，最多允许 5 轮迭代，确保安全性与收敛性。

**AI-IDE 联动交互：** 前端将 Monaco Editor 集成为在线 IDE，支持将当前题目上下文与用户源代码一键同步至 AI 助手的对话上下文；AI 生成的修复方案可一键回写至编辑器，形成"诊断—修复—验证"的闭环工作流。

**技术选型与开发实施：** 后端基于 Java 21 + Spring Boot 3.2 + MyBatis-Plus 构建，数据持久层采用 MySQL 8.0（配置主从复制，主库负责写入、从库分担读查询），热点数据经 Redis 缓存加速，测试数据文件存储于 MinIO 对象存储。前端采用 React 19 + TypeScript + Ant Design + TailwindCSS 构建单页应用（SPA），代码编辑器基于 Monaco Editor 实现语法高亮与智能补全。消息中间件选用 RabbitMQ，部署层使用 Docker Compose 编排各服务容器并以 Nginx 作为反向代理。

开发过程中，后端按用户、题目、评测、比赛、权限等功能域分模块迭代，其中评测模块涉及异步消息消费、沙箱 HTTP 交互与 WebSocket 推送三者的协调，为实现复杂度最高的子系统。前后端联调阶段通过反复的接口测试与性能调优，确保系统在 50 并发提交场景下各模块功能正确且响应时间可控。

**技术栈总览：** 后端 Java 21 + Spring Boot 3.2 + MyBatis-Plus + MySQL 8.0 + Redis + RabbitMQ + MinIO；前端 React 19 + TypeScript + Ant Design + TailwindCSS + Monaco Editor；部署 Docker Compose + Nginx。

# 2 系统开发环境

## 2.1 Java简介

Java 由 Sun Microsystems 于 1995 年发布，是一门面向对象的通用编程语言，依托 JVM（Java Virtual Machine）实现"一次编写，处处运行"的跨平台特性，长期占据服务端开发的主流地位。

本系统选用 Java 21（2023 年 9 月发布的 LTS 长期支持版本）作为后端开发语言，其相较于 Java 17 的核心升级在于**虚拟线程（Virtual Threads, JEP 444）**的正式转正。传统 Java 线程与操作系统内核线程一一映射，线程创建与上下文切换的开销使得单 JVM 实例难以承载超过数千个并发连接；而虚拟线程由 JVM 在用户态自行调度，栈空间按需分配，单进程可并发管理数十万乃至百万级轻量级线程，尤其适用于 I/O 密集型的评测队列消费场景。本项目在 application.yml 中通过 `spring.threads.virtual.enabled: true` 全局启用虚拟线程，使 RabbitMQ 消费者与 WebSocket 推送等阻塞式 I/O 操作自动运行于虚拟线程之上，有效提升了吞吐量。

此外，Java 21 还引入了 Record 类型、模式匹配（Pattern Matching）、密封类（Sealed Classes）等语言特性，有助于减少样板代码；ZGC 与 G1 垃圾收集器的持续优化也使得 GC 停顿时间进一步缩短。Spring Boot 3.x 系列要求 Java 17 及以上版本，选用 Java 21 既满足框架约束，又能充分利用最新的语言与运行时改进。

## 2.2 Python AI 代理层与流式响应技术

Python 在大语言模型生态中拥有最为完善的 SDK 支持——OpenAI、Google Generative AI、zhipuai 等厂商均优先发布 Python 客户端库，且 FastAPI 框架原生支持异步 I/O 与流式响应。基于此，本系统将 AI 调用逻辑从 Java 主体中剥离，构建了一个独立运行的 Python 代理服务（ai-proxy），其核心设计体现在三个层面：

**解耦架构：** Java 后端（AiController）仅负责将用户的聊天消息、模型名称与鉴权参数封装为 HTTP 请求，经内网转发至 Python 代理层的 /v1/chat/stream 或 /v1/chat/stream/tools 端点。两者通过 HTTP + SSE 协议通信，业务逻辑与模型调用完全分离，新增模型厂商仅需在代理层注册对应的 Provider 实现类，无需改动 Java 端代码。

**多模型路由与适配：** 代理层内部的 provider_registry 模块维护了模型名到厂商 Provider 的映射关系。调用方传入形如 "deepseek-chat"、"gemini-2.0-flash" 的模型标识后，normalize_model_name 函数将其规范化，get_provider 函数据此路由至对应厂商的异步 stream 方法。各 Provider 封装了差异化的 API 鉴权、请求格式与错误处理逻辑，对上层暴露统一的异步迭代器接口。

**SSE 流式响应：** 大语言模型的推理延迟通常在数秒至十余秒量级，若采用同步 HTTP 请求，用户需等待完整响应生成后方可看到结果。代理层采用 SSE（Server-Sent Events）协议，将模型生成的文本以 `data: {"type":"content","delta":"..."}` 的格式逐 chunk 推送至前端，形成逐字输出的流式交互效果。

## 2.3 MySQL数据库

MySQL 是 Oracle 公司维护的开源关系型数据库管理系统，凭借其成熟的 InnoDB 存储引擎、活跃的社区生态以及广泛的框架适配性，长期作为 Web 应用后端的首选数据持久层。本系统选用 MySQL 8.0 版本，相较于 5.7 版本，其在以下方面具有显著改进：

- InnoDB 读写混合场景下的吞吐量提升约 2 倍，得益于 redo log 优化与自适应哈希索引的改进
- 原生 JSON 列类型支持，可存储题目的扩展配置信息并直接在 SQL 中进行路径查询
- 引入窗口函数（Window Functions），使排名统计、累计计算等分析型查询的编写复杂度显著降低
- DDL 操作具备原子性保障，避免了表结构变更中途失败导致的元数据不一致

在数据层架构方面，本系统配置了一主一从两个 MySQL 实例（主库端口 3308，从库端口 3309），采用基于 binlog 的异步复制实现数据同步。OJ 系统的访问模式呈典型的读多写少特征——题目浏览、提交记录查询等读请求量远高于代码提交产生的写请求——因此将读查询路由至从库可有效分摊主库的 I/O 压力，降低写操作的锁竞争概率。实现层面，项目通过 dynamic-datasource-spring-boot-starter 组件（配置 `spring.datasource.dynamic.primary: master`）实现数据源的动态路由，Service 层方法可通过 @DS 注解显式指定目标数据源，业务代码对读写分离逻辑基本无感知。

## 2.4 B/S结构

B/S（Browser/Server）架构是一种以浏览器为客户端、以 Web 服务器为核心的分布式计算模型，与传统 C/S（Client/Server）架构相比，其显著优势在于客户端零安装——用户仅需通过浏览器访问系统 URL 即可使用全部功能，系统升级与缺陷修复只需在服务端完成部署，客户端无需任何更新操作。

B/S 架构与 OJ 系统的需求特征具有较高的契合度：

- **零部署成本：** 用户无需安装专用客户端软件，降低了系统的推广门槛
- **跨平台兼容：** 基于 Web 标准的前端可运行于 Windows、macOS、Linux 及移动端浏览器
- **集中式维护：** 业务逻辑与数据集中于服务端，便于统一管理和持续迭代

B/S 架构对网络连通性的依赖较强，且部分操作系统级功能（如本地文件系统深度访问）受浏览器沙箱限制，但这些约束对于以代码编辑与在线评测为核心的 OJ 系统而言影响甚微。

## 2.5 Spring Boot框架

Spring Boot 是 Pivotal 团队在 Spring Framework 基础上构建的快速开发框架，其设计哲学为"约定优于配置"（Convention over Configuration）——通过自动配置（Auto-Configuration）机制与起步依赖（Starter Dependencies）大幅减少了传统 Spring 项目中繁琐的 XML 配置工作，使开发者能够以注解驱动的方式快速搭建生产级应用。

本系统采用 Spring Boot 3.2.5 版本。3.x 系列相较于 2.x 的核心变化在于底层从 Java EE 迁移至 Jakarta EE 规范，包名由 `javax.*` 统一变更为 `jakarta.*`，同时要求 Java 17 及以上运行时环境。项目中涉及的 Spring 生态组件包括：

- **Spring MVC：** 基于 DispatcherServlet 的请求分发机制，承载 RESTful API 的路由与参数绑定
- **MyBatis-Plus：** 在 MyBatis 基础上提供 Lambda 查询构造器与通用 CRUD 接口，减少 SQL 手写量
- **Spring AMQP：** 封装 RabbitMQ 的连接管理、消息序列化与监听器容器，支撑评测任务的异步调度
- **Spring WebSocket：** 提供 TextWebSocketHandler 抽象，用于评测结果的服务端主动推送
- **Spring Data Redis：** 提供 RedisTemplate 抽象，用于热点数据缓存与验证码等临时状态的存取

此外，Spring Boot 内嵌 Tomcat 服务器，编译产物为可独立运行的 Fat JAR，通过 `java -jar` 即可启动，结合 Docker 容器化部署可进一步简化运维流程。

### 3.2.2 用户登录流程

系统的身份认证采用 JWT（JSON Web Token）无状态方案，其流程如下：

1\. 用户在登录页输入邮箱与密码

2\. 前端将凭据以 POST 方式提交至 /api/user/login 接口

3\. 后端从数据库中检索对应用户记录，使用 BCrypt 算法对提交的明文密码与存储的哈希值进行比对验证

4\. 验证通过后，后端签发 JWT Token（有效期 7 天），Token 载荷中包含用户 ID 与角色信息

5\. 前端将 Token 存储于 localStorage，后续每次 HTTP 请求均在请求头中附加 `Authorization: Bearer <token>` 字段

6\. 后端的认证过滤器（Filter）在请求到达 Controller 之前拦截并校验 Token 的签名与有效期，校验失败则返回 401 状态码

选择 JWT 而非传统 Session 机制的原因在于：JWT 将认证状态编码于 Token 自身，服务端无需维护会话存储，天然适配水平扩展与分布式部署场景。

### 3.2.3 代码评测流程

代码评测是系统的核心功能，其流程涉及前端提交、消息队列异步调度、沙箱执行与结果推送四个环节：

图3.3 后端评测流程图

用户提交代码后，后端 SubmissionController 将提交信息（代码、语言、题目 ID 等）封装为 JudgeInfo 对象并发布至 RabbitMQ 的 submissionQueue。JudgeConsumer 监听该队列，从消息中反序列化 JudgeInfo，通过 JudgeStrategyFactory 根据语言字段获取对应的评测策略实例（如 CppJudgeStrategy），调用其 judge 方法完成编译、批量执行与输出比对。评测完成后，结果经 WebSocket 通道实时推送至前端。

采用消息队列进行异步解耦的核心原因在于：编译与执行操作的耗时从数百毫秒到数秒不等，若以同步方式处理，用户提交后需持续等待评测完成，HTTP 连接长时间占用将显著降低系统的并发处理能力。异步方案下，提交请求即时返回确认，评测任务在后台由消费者按序处理，评测状态通过 WebSocket 增量推送，用户感知的响应延迟仅为消息入队的时间。

图3.4 代码评测流程图

### 3.2.4 比赛流程

比赛模块的业务流程遵循 ACM-ICPC 赛制规范，具体步骤如下：

1\. 管理员创建比赛，配置标题、起止时间，可选设置访问密码

2\. 管理员为比赛添加题目，系统按字母顺序自动编号（A、B、C...）

3\. 比赛开始前，用户可进行报名操作（受密码保护的比赛需输入正确密码方可报名）

4\. 比赛开始后，已报名用户方可查看题目内容并提交代码

5\. 每次提交均经过完整的评测流程，系统记录提交时间、评测结果（Accepted 或其他状态）与罚时

6\. 排名榜实时更新，排序规则为：首先按 Accepted 题目数降序排列，题数相同时按累计罚时升序排列

7\. 比赛结束时刻，排名自动锁定，不再接受新提交

罚时计算采用 ACM-ICPC 标准规则：对于每道 Accepted 的题目，罚时 = 从比赛开始至该题首次 Accepted 的分钟数 + 该题此前 Wrong Answer 次数 × 20 分钟。此规则旨在同时衡量解题速度与正确率。

# 4 系统设计

## 4.1 系统概述

本系统采用前后端分离架构，前端（React SPA）与后端（Spring Boot）作为两个独立部署的工程，通过 RESTful API 经 Nginx 反向代理进行通信。架构设计围绕以下四个质量属性展开：

**稳定性：** 评测任务经 RabbitMQ 持久化队列异步调度，即使消费者进程异常重启，队列中未确认的消息仍会被重新投递，从而避免评测任务丢失。submissionQueue 配置了 `durable=true` 参数确保队列在 Broker 重启后依然存在。

**性能：** 系统需在比赛高峰期承载数十名用户的并发提交。热点数据（如题目信息、测试用例）经 Redis 缓存加速读取，MySQL 主从复制将读查询分流至从库，RabbitMQ 消费者的 `prefetchCount=1` 配置确保单个消费者顺序处理评测任务，避免资源争用。

**安全性：** 用户提交的代码在 Go-Judge 沙箱中执行，沙箱通过 Linux cgroup 限制 CPU 时间与内存用量，通过 namespace 隔离文件系统与网络栈，有效防御 fork 炸弹、敏感文件读写及非授权网络请求等攻击向量。

**可维护性：** 后端遵循 Controller-Service-Mapper 三层架构，各层职责边界清晰；评测模块通过策略模式（JudgeStrategy 接口 + JudgeStrategyFactory 工厂）实现多语言评测逻辑的解耦，新增语言支持仅需实现接口并注册至 Spring 容器。

## 4.2 系统结构设计

### 4.2.1 分层架构

后端代码采用经典的 MVC 分层架构。Controller 层仅负责 HTTP 请求参数的接收、校验与响应封装，不包含任何业务逻辑；Service 层承载核心业务规则与事务管理；Mapper 层通过 MyBatis-Plus 与数据库交互，执行持久化操作。各层之间通过接口抽象解耦，上层仅依赖下层的接口定义而非具体实现，修改单层逻辑时对其他层的影响可控。

图4.2 系统功能结构图

### 4.2.2 后端目录结构

后端工程以 Spring Boot 标准结构组织，根包为 com.example.vnollxonlinejudge，主要包含以下模块：

**注解与切面层：** annotation 包定义了 RequirePermission、RequireRole、RateLimiter 三个自定义注解，分别用于接口级权限校验、角色校验与请求频率限制；aop 包通过 Spring AOP 切面拦截被上述注解标注的方法，在方法执行前完成鉴权与限流判断。

**配置层：** config 包集中管理 Redis 连接池、RabbitMQ 队列与交换机声明、MinIO 客户端初始化、WebSocket 端点注册等基础设施配置，各配置类均以 @Configuration 注解标注，由 Spring Boot 自动装配机制加载。

**消息消费层：** consumer 包包含 JudgeConsumer 与 NotificationConsumer 两个消息监听器，前者负责从 submissionQueue 中消费评测任务并调用判题策略执行评测，后者处理系统通知的异步投递。

**控制器层：** controller 包按业务域划分为 UserController、ProblemController、SubmissionController、CompetitionController、AiController 等，每个控制器仅负责请求参数校验与响应封装，不包含业务逻辑。

**评测引擎层：** judge 包是评测子系统的核心，包含 JudgeStrategy 策略接口、JudgeStrategyFactory 策略工厂以及各语言的策略实现类（如 CppJudgeStrategy、JavaJudgeStrategy 等），通过策略模式实现多语言评测逻辑的解耦。

**过滤器层：** filter 包实现 JWT 认证过滤器，在请求到达控制器之前完成 Token 签名校验与用户上下文注入。

**数据模型层：** model 包按职责细分为 entity（数据库实体类）、dto（数据传输对象）和 vo（视图对象）三个子包，实现各层之间的数据隔离。

**持久化与服务层：** mapper 包定义 MyBatis-Plus 的 Mapper 接口，service 包承载核心业务逻辑，其 impl 子包提供具体实现。producer 包封装 RabbitMQ 消息发布逻辑，utils 包提供雪花 ID 生成、密码加盐哈希等通用工具，websocket 包实现评测结果的服务端主动推送。

### 4.2.3 前端目录结构

前端工程基于 Vite + React + TypeScript 构建，源码根目录为 frontend/src，主要包含以下模块：

**入口与路由：** main.tsx 为应用入口，App.tsx 配置全局路由表，采用 React Router 实现页面间的声明式导航。index.css 定义全局样式变量与基础重置样式。

**公共组件层：** components 目录封装了多个跨页面复用的 UI 组件，包括 AIAssistant（AI 助手悬浮聊天窗）、Auth（登录注册弹窗）、CodeEditor（基于 Monaco Editor 的代码编辑器封装，支持多语言语法高亮）、Layout（顶部导航栏与页面布局框架）以及 ParticleBackground（首页 Canvas 粒子动画背景）。

**页面层：** pages 目录按功能域组织各页面组件，主要包括 Home（首页，展示平台统计数据与数字滚动动画）、Problems（题目列表，支持分页与关键字检索）、ProblemDetail（做题页面，集成代码编辑器与提交控制）、Competitions 与 CompetitionDetail（比赛列表与比赛详情）、Submissions（提交记录，WebSocket 实时状态更新）、Profile（个人中心，含做题统计与资料修改）以及 Admin 子目录（管理后台，涵盖用户、题目、比赛、题解等管理模块）。

**工具与类型层：** hooks 目录定义自定义 React Hook（如 useJudgeWebSocket 用于评测结果的 WebSocket 监听），utils 目录包含 Axios 请求封装（统一拦截、Token 注入）与登录状态管理工具函数，types 目录集中维护 TypeScript 接口与类型定义，确保前后端数据契约的一致性。

## 4.3 数据库设计

### 4.3.1 设计原则

数据库设计遵循以下原则：

- **第三范式（3NF）：** 消除传递依赖，将具有独立语义的实体拆分为独立表（如用户表与角色表分离），以减少数据冗余并降低更新异常的风险
- **索引策略：** 对高频查询字段（如 problem 表的 difficulty、submission 表的 uid 与 pid）建立 B+Tree 索引，同时避免在低选择性字段上滥建索引以控制写操作的额外开销
- **命名规范：** 表名采用小写下划线命名法，主键统一命名为 id（自增 BIGINT），时间戳字段统一使用 create_time / update_time 并设置默认值与自动更新

## 6.1 系统测试的意义

系统测试是软件开发生命周期中不可缺省的质量保障环节，其核心目标包括：

- 在系统交付前尽早暴露功能缺陷与逻辑漏洞，降低线上故障风险
- 验证各功能模块的实现是否与需求规格一致
- 覆盖边界条件与异常路径，确保系统在非预期输入下的鲁棒性
- 评估系统在目标负载下的性能指标是否达到预设阈值

对于 OJ 系统而言，测试的重要性尤为突出：评测模块直接执行用户提交的任意代码，若沙箱隔离存在缺陷，恶意代码可能导致宿主机资源耗尽甚至安全入侵，因此评测安全性的验证必须作为测试工作的重中之重。

图6.1 测试与纠错流程图

## 6.2 测试方法

本系统的测试工作采用以下四类方法：

**功能测试（黑盒测试）：** 不关注内部实现逻辑，仅根据需求规格验证系统输入与输出的正确性。例如登录功能测试中，分别验证正确凭据下的成功登录与错误密码下的拒绝响应，确认系统行为与预期一致。

**接口测试：** 使用 Postman 工具对后端 RESTful API 逐一发起请求，校验响应的 HTTP 状态码、JSON 数据结构及异常场景下的错误码与提示信息是否符合接口契约。

**评测正确性与安全性测试：** 针对评测模块设计覆盖性测试用例，包括：

- 正确解法代码（预期 Accepted）
- 逻辑错误代码（预期 Wrong Answer）
- 死循环代码（预期 Time Limit Exceeded）
- 超大内存申请代码（预期 Memory Limit Exceeded）
- 段错误 / 除零错误代码（预期 Runtime Error）
- 语法错误代码（预期 Compile Error）
- 恶意代码（fork 炸弹、文件系统读写、网络外连，预期被沙箱拦截）

**并发压力测试：** 使用 JMeter 工具模拟多用户并发提交场景，评估系统在高负载下的吞吐量与响应延迟。

# 结论

本课题围绕高校编程教学中对智能化、安全化在线评测系统的需求，设计并实现了一套基于 Spring Boot + React 的 AI 驱动型 Online Judge 系统。经过需求调研、方案设计、系统开发与测试验证四个阶段的工作，系统最终实现了以下核心功能：

- 用户注册与登录（基于邮箱验证码与 JWT 无状态认证）
- 题库管理（题目 CRUD、多级难度、算法标签体系）
- 代码在线评测（支持 C/C++/Java/Python/JavaScript/Go 六种语言，Go-Judge 沙箱隔离，WebSocket 结果实时推送）
- 比赛系统（报名、ACM-ICPC 赛制排名与罚时计算）
- 权限控制（基于 RBAC 模型的细粒度访问控制）
- AI 智能助手（多大语言模型接入、Function Calling 工具调用、SSE 流式响应）

在技术层面，本项目将消息队列异步调度、MySQL 主从读写分离、Redis 缓存加速、内核级沙箱隔离等多项技术有机整合，系统复杂度相较于传统 CRUD 应用有显著提升。Go-Judge 沙箱的集成与 WebSocket 实时推送的调试过程中遇到了较多工程难点，前者涉及编译产物的文件 ID 管理与批次化执行策略，后者需处理跨域配置与会话映射的一致性问题。

系统目前仍存在以下可改进之处：

- 编程语言支持范围有限，后续可扩展 Rust、Go 等更多语言
- 缺少代码相似度检测功能，比赛场景下的作弊识别能力不足
- 大规模并发场景（千人级比赛）下的性能表现尚未经过充分验证
- 部分页面的交互细节与响应式适配仍有优化空间

后续工作可考虑引入代码查重算法（如基于 AST 的结构化比对）、水平扩展评测消费者实例以提升并发吞吐量，以及将 AI 助手的工具调用能力扩展至代码调试与自动化测试生成等方向。

**代码 A.22** AI 助手——Python 代理 SSE 封装与按模型流式输出

```python
# ai-proxy/main.py（节选）
def sse_event(payload: dict) -> bytes:
    return f"data: {json.dumps(payload, ensure_ascii=False)}\n\n".encode("utf-8")

@app.post("/v1/chat/stream")
async def chat_stream(request: ChatStreamRequest) -> StreamingResponse:
    provider = get_provider(request.model)
    actual_model = normalize_model_name(request.model)
    normalized = request.model_copy(update={"model": actual_model})
    async def event_stream() -> AsyncIterator[bytes]:
        yield sse_event({"type": "meta", "provider": provider.provider_name, "model": actual_model})
        async for chunk in provider.stream(normalized):
            if chunk:
                yield sse_event({"type": "content", "delta": chunk})
        yield sse_event({"type": "done"})
    return StreamingResponse(
        event_stream(),
        media_type="text/event-stream",
        headers={"Cache-Control": "no-cache", "Connection": "keep-alive", "X-Accel-Buffering": "no"},
    )
```
