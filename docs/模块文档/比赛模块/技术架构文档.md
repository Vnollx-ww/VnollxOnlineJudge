# 比赛模块技术架构文档

## 模块职责

比赛模块负责比赛元数据、参赛关系、比赛题目、团队信息、比赛提交统计、排行榜和防作弊数据的组织。

## 代码入口

- **用户比赛接口**：`CompetitionController`
- **管理比赛接口**：`AdminCompetitionController`
- **防作弊接口**：`AdminCompetitionAntiCheatController`
- **评测入口依赖**：`JudgeService`

## 数据模型

- **比赛表**：保存比赛名称、时间、类型、赛制、状态等信息。
- **比赛题目表**：维护比赛与题目的关系、顺序和统计。
- **比赛用户表**：保存个人赛用户或团队赛队长维度的统计。
- **比赛队伍表**：保存团队名称、邮箱、手机号和女子组等报名信息。
- **比赛队员表**：保存队员真实姓名等文本信息。
- **提交表**：保存比赛上下文中的提交记录。

## 核心流程

```text
管理员创建比赛并绑定题目
-> 用户或团队完成报名
-> 比赛开始后用户提交代码
-> 评测模块返回结果
-> 比赛模块按赛制聚合统计
-> 排行榜读取统计并展示
```

## 团队赛统计规则

团队赛中，队长一定有平台账号，队员不要求有平台账号。团队信息按报名邮箱识别，并在统计时归属到队长平台账号维度。

```text
提交用户
-> 根据比赛和用户识别所属队伍
-> 通过队伍邮箱匹配队长账号邮箱
-> 聚合到 competition_user 中的队长账号
-> 排行榜展示时补充队伍信息
```

## 外部依赖

- **评测模块**：提供比赛提交结果。
- **Redis**：用于 ACM 赛制实时榜单、通过数和罚时等缓存。
- **MySQL**：持久化比赛、队伍、报名、题目和提交数据。
- **文件导入能力**：团队赛报名 Excel 导入。

## 缓存与异步

- **ACM 榜单缓存**：通过 Redis 维护通过数、罚时和排序。
- **OI 结算**：比赛结束后统一投递保存的提交进行评测，再聚合得分。
- **防作弊事件**：可异步采集和聚合风险数据。

## 安全与权限

- **比赛时间校验**：提交前必须校验比赛是否开放。
- **参赛资格校验**：提交前校验用户是否属于比赛或队伍。
- **管理权限校验**：比赛配置和队伍导入仅管理员可执行。

## 异常处理

- **未报名提交**：返回参赛资格错误。
- **比赛未开始或已结束**：阻止提交或保存。
- **队伍归属失败**：团队赛提交应返回明确业务错误，避免错误计榜。
- **榜单缓存不一致**：必要时以数据库聚合结果进行修正。

## 扩展点

- **新赛制**：可扩展 IOI、ICPC 封榜、滚榜等规则。
- **报名审核**：支持报名审批、缴费状态和队伍资料校验。
- **高级防作弊**：支持代码相似度、行为时序和设备风险分析。
