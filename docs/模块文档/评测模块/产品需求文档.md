# 评测模块产品需求文档

## 模块定位

评测模块负责接收代码提交、调度评测任务、返回评测结果，并为题库、比赛、练习、排名和用户统计提供判题数据。

## 用户角色

- **普通用户**：提交代码并查看评测状态和结果。
- **参赛用户**：在比赛上下文中提交代码，结果影响榜单。
- **管理员**：查看提交记录、排查评测异常和管理评测配置。

## 核心场景

- **普通提交**：用户在题目页提交代码，实时查看结果。
- **比赛提交**：用户在比赛题目页提交代码，结果进入比赛统计。
- **提交列表**：用户和管理员查看提交历史与详情。
- **队列反馈**：提交后展示排队、评测中、完成或失败状态。

## 功能需求

- **多语言提交**：支持多种编程语言及对应编译运行配置。
- **异步评测**：提交后进入队列，由评测消费者异步执行。
- **实时推送**：通过实时通道推送评测进度和最终结果。
- **结果展示**：展示状态、耗时、内存、错误信息和测试点统计。
- **评测机调度**：支持多评测节点和容量控制。

## 业务规则

- **登录提交**：提交代码需要登录。
- **语言限制**：只允许提交系统支持的语言。
- **比赛约束**：比赛提交需校验比赛时间、参赛资格和赛制规则。
- **结果可信**：评测结果以沙箱执行和输出比较结果为准。

## 页面与交互

- **题目详情页**：代码编辑器、语言选择、提交按钮、结果区域。
- **提交列表页**：筛选提交记录，查看状态和详情。
- **管理后台提交页**：按用户、题目、语言、状态检索提交。

## 权限要求

- **本人可见**：用户可查看自己的提交详情。
- **公开列表**：普通题目提交列表可按系统策略展示部分信息。
- **管理权限**：管理员可查看全量提交和错误信息。

## 非功能需求

- **隔离性**：用户代码必须在沙箱隔离环境运行。
- **稳定性**：评测失败不能影响后端主服务稳定性。
- **可观测性**：评测链路应具备日志、状态和错误信息。

## 验收标准

- **提交可完成**：合法代码能够进入队列并返回结果。
- **错误可定位**：编译错误、运行错误、超时、内存超限有明确状态。
- **比赛统计正确**：比赛提交能正确影响榜单和题目统计。
