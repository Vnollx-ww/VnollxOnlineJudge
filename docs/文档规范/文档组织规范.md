# 文档组织规范

## 目标

让文档结构能够稳定支撑产品、开发、测试、部署和后续演进，避免所有内容堆在根目录或单个大文档中。

## 目录分层

| 目录 | 内容范围 |
|:-----|:---------|
| `总览` | 系统级产品需求、系统级技术架构、跨模块共性说明 |
| `模块文档` | 按业务模块维护产品需求文档和技术架构文档 |
| `运维部署` | 部署、环境、中间件、监控、排障、发布流程 |
| `演进规划` | 未来版本、重构计划、赛制扩展、技术路线 |
| `归档` | 论文、历史方案、过期材料、临时调研 |

## 模块文档要求

每个模块目录至少包含两个文档：

- **产品需求文档**：描述模块面向用户和业务的需求。
- **技术架构文档**：描述模块在系统内如何实现、依赖哪些服务、核心数据如何流转。

## 产品需求文档模板

```markdown
# 模块名称产品需求文档

## 模块定位

## 用户角色

## 核心场景

## 功能需求

## 业务规则

## 页面与交互

## 权限要求

## 非功能需求

## 验收标准
```

## 技术架构文档模板

```markdown
# 模块名称技术架构文档

## 模块职责

## 代码入口

## 数据模型

## 核心流程

## 外部依赖

## 缓存与异步

## 安全与权限

## 异常处理

## 扩展点
```

## 命名约定

- **目录名使用中文**：不要使用英文目录名，也不要使用数字编号目录名。
- **文件名尽量中文**：模块文档统一使用 `产品需求文档.md` 和 `技术架构文档.md`。
- **同类文档同名**：不同模块下允许使用相同文件名，依靠目录区分模块。
- **索引必须更新**：新增模块或移动文档后，必须同步更新 `README.md` 和 `模块文档/README.md`。

## 归档规则

- **仍在维护的全局内容**：移动到 `总览`、`模块文档` 或 `运维部署`。
- **仅用于参考的历史材料**：移动到 `归档`。
- **演进中的方案**：放到 `演进规划`，落地后再拆分到对应模块文档。
