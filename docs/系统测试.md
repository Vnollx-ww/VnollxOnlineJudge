# 6 系统测试

## 6.1 系统测试的意义

测试这个环节不能省。之前做项目有过惨痛教训，自己觉得写完了没问题，一上线全是 bug，用户体验极差。

测试的意义在于：

- 提前发现问题，避免上线后出丑
- 验证功能是否符合需求
- 检查边界情况和异常处理
- 评估性能是否达标

OJ 系统测试尤其重要，因为评测功能涉及代码执行，一旦有漏洞后果很严重（比如用户代码把服务器搞崩了）。

## 6.2 测试方法

测试主要用了这几种方法：

**功能测试（黑盒）**：不管代码怎么实现的，只看功能对不对。比如测试登录功能，输入正确的邮箱密码，应该登录成功；输入错误的密码，应该提示错误。

**接口测试**：用 Postman 测后端 API（本系统统一前缀 `/api/v1`），检查返回格式、状态码、异常处理。

**评测测试**：重点测各种边界情况——正常 AC 的代码；答案错误的代码（WA）；死循环代码（TLE）；申请大数组的代码（MLE）；段错误/除零错误（RE）；语法错误（CE）；恶意代码（fork 炸弹、读写文件、网络请求）。

**并发测试**：用 JMeter 模拟多用户同时提交，看系统能不能扛住。

## 6.3 测试分析

**测试结果汇总（对应原表 6.1，此处改为文字叙述）**：用户登录注册模块共 12 条用例，全部通过，通过率 100%。题目 CRUD 15 条全部通过。代码评测 20 条全部通过。比赛功能 16 条全部通过。权限控制 10 条全部通过。

**性能测试结果**：首页加载平均约 1.2 秒；题目详情 API 平均约 180ms；从提交代码到收到评测结果平均约 3～5 秒（含编译与运行）；50 并发提交时系统表现正常，无明显延迟。

**安全测试**：fork 炸弹在沙箱内被限制，进程数超限会被终止；读宿主机敏感路径在隔离环境下无法得逞；网络访问在沙箱侧被禁止。

**总体评价**：功能基本完善，性能满足小规模使用（校内几百人），安全性有基本保障。

### 6.3.1 代表性测试用例（结合本系统逐条说明）

下面六条与 6.2 中的黑盒、接口、评测与安全测试相呼应，正文里按条展开即可，不必再制表。

**（1）登录成功（用户登录注册 / 接口）**  
在 Postman 或前端页面使用已注册账号，请求 `POST /api/v1/user/login`，请求体为正确邮箱与密码。预期：返回业务成功与 JWT；随后在请求头携带 `Authorization: Bearer <token>` 调用 `GET /api/v1/user/profile`，能拿到当前用户资料。用于验证鉴权链路与 Token 签发是否正常。

**（2）登录失败（用户登录注册 / 异常）**  
同一邮箱输入错误密码再次调用登录接口。预期：登录失败，提示密码错误，且不返回可用于访问受保护接口的有效 Token。用于验证错误凭据不会误放行。

**（3）题目详情查询（题目模块 / 接口与展示）**  
对已发布题目调用 `GET /api/v1/problem/get`，通过题号参数查询。预期：返回题目标题、时限、内存限制等结构化信息；题目不存在或参数非法时，返回与系统约定一致的错误提示，前端不崩溃。用于验证题库对外接口与数据完整性。

**（4）管理端创建题目权限（权限控制）**  
使用**普通用户**身份（仅有常规做题权限、无题目创建权限）携带 Token 调用 `POST /api/v1/admin/problem/create`。预期：请求被拒绝（如无权限或统一业务错误码），题库中不应因该请求产生新题。用于验证 RBAC 与 `@RequirePermission` 一类保护是否生效。

**（5）正解提交通过（代码评测）**  
已登录且具备提交权限的用户，对某题提交能通过全部测试点的源代码，请求 `POST /api/v1/judge/submit`，请求体包含代码、语言、题号、时限与内存限制等字段（练习场景下比赛 id 可为 0）。预期：接口立即返回成功及提交单标识（如 snowflakeId）；经消息队列与评测 Worker 处理后，该条提交终态为「答案正确」。用于验证「提交—入队—评测—回写」整条链路。

**（6）恶意或极端代码（评测安全与资源限制）**  
在同一套评测链路上，分别提交死循环、超大内存申请或 fork 炸弹等典型恶意/极端代码。预期：得到「时间超出限制」「内存超出限制」或沙箱拦截等终态之一，评测 Worker 与主业务进程保持稳定，不出现整机卡死或宿主机被拖垮。用于验证沙箱与资源上限是否真正起作用。
