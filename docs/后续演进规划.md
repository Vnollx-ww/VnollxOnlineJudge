# Vnollx 在线评测系统 - 后续演进规划

## 1. 文档说明

本文档用于记录 Vnollx Online Judge 的后续功能演进方向，重点包括比赛赛制扩展与题目评测模块重构。

本文档内容属于后续规划与设计方案，不代表当前版本已经全部实现。

---

## 2. 演进目标

后续系统演进主要围绕以下方向展开：

- **比赛赛制扩展**：在现有 ACM 赛制基础上，引入 OI 与 IOI 赛制。
- **评测逻辑重构**：将当前集中在评测流程中的硬编码判断拆分为多个评测模式类。
- **策略化扩展**：通过类似策略模式的方式，让不同题目评测模式拥有独立的业务逻辑。
- **前端组件化治理**：拆分冗余页面代码，沉淀可复用业务组件、Hooks 与工具层。
- **前端性能优化**：补充路由懒加载、渲染优化、请求缓存、资源拆包等基础性能能力。
- **Next.js 迁移预研**：评估从当前 Vite SPA 向 Next.js 架构迁移的收益、成本与落地路径。
- **降低耦合**：减少评测主流程对具体题型、输出比较规则、得分规则的直接依赖。
- **便于扩展**：后续新增题型、赛制、计分方式时，尽量只新增策略类，而不是修改核心评测流程。

---

## 3. 比赛赛制扩展规划

### 3.1 当前赛制

当前比赛系统主要按 ACM 赛制组织：

- 提交结果以是否通过为核心。
- 题目状态主要关注 AC、尝试次数、首杀等信息。
- 排名通常按解题数、罚时等规则计算。

### 3.2 OI 赛制

后续计划支持 OI 赛制，用于更适合阶段性练习、选拔赛、校内训练等场景。

OI 赛制核心特点：

- 每道题有独立分值。
- 一道题可以获得部分分。
- 排名主要按总分排序。
- 通常不强调实时罚时，也不一定展示提交失败惩罚。
- 比赛结束后可以统一公布成绩，也可以按配置控制是否赛中展示分数。

计划支持能力：

- 比赛配置中增加赛制字段，例如 `ACM`、`OI`、`IOI`。
- 比赛题目支持配置分值。
- 提交记录保存单次提交得分。
- 排行榜按用户或队伍的各题最高分汇总总分。
- 支持比赛期间隐藏或展示分数。

### 3.3 IOI 赛制

后续计划支持 IOI 赛制，用于更精细的测试点计分场景。

IOI 赛制核心特点：

- 每道题由多个测试点或测试点组组成。
- 每个测试点或测试点组拥有独立分值。
- 一次提交可以根据通过的测试点获得部分分。
- 排名一般按各题最高得分求和。
- 更强调测试点级别的评测结果与得分明细。

计划支持能力：

- 测试数据支持测试点分值或测试点组分值配置。
- 评测结果保存每个测试点的状态、耗时、内存与得分。
- 题目最终得分由测试点或测试点组得分聚合得到。
- 排行榜按题目最高得分或最新有效得分进行汇总。
- 管理端支持查看单题得分明细与测试点通过情况。

## 4. 评测模块重构规划

### 4.1 当前问题

当前评测流程中，部分题型判断、输出比较、特殊评测、浮点误差等逻辑集中在评测主流程中。

随着后续引入 OI、IOI、更多题型和更多计分规则，如果继续在主流程中增加条件判断，会带来以下问题：

- 主流程越来越复杂。
- 不同评测模式之间逻辑耦合。
- 新增题型容易影响已有题型。
- 单元测试和问题定位难度上升。
- 计分逻辑和运行逻辑边界不清晰。

### 4.2 重构目标

评测模块后续重构目标：

- 将评测执行流程与具体评测模式解耦。
- 每一种评测模式由独立类负责。
- 主评测流程只负责任务编排、资源调度、状态流转和结果落库。
- 具体模式类负责输出比较、Checker 调用、测试点计分、结果聚合等业务逻辑。
- 支持后续按需新增 `standard`、`special`、`oi`、`ioi` 等模式。

## 5. 策略模式设计思路

### 5.1 核心思想

后续可以将题目评测模式抽象为统一接口，每一种评测模式实现一个独立策略类。

主流程根据题目的 `judgeMode`、比赛的 `contestMode` 或其他配置，选择对应策略执行。

整体思路类似策略模式：

```text
JudgeConsumer
    → JudgeService
        → JudgeModeResolver
            → JudgeModeStrategy
                → StandardJudgeStrategy
                → SpecialJudgeStrategy
                → OiJudgeStrategy
                → IoiJudgeStrategy
```

### 5.2 建议抽象

可以设计统一的评测策略接口：

```text
JudgeModeStrategy

职责：
- 判断是否支持当前评测模式。
- 执行对应模式下的评测逻辑。
- 返回统一的评测结果对象。
```

统一入参可以包含：

- 提交信息。
- 题目信息。
- 比赛信息。
- 语言配置。
- 测试数据信息。
- Go-Judge 端点上下文。

统一出参可以包含：

- 总体状态。
- 总得分。
- 最大耗时。
- 最大内存。
- 编译信息。
- 测试点明细。
- Checker 输出。
- 错误信息。

## 6. 评测策略拆分建议

### 6.1 StandardJudgeStrategy

用于标准题评测。

职责：

- 编译用户代码。
- 运行所有测试点。
- 比较用户输出和标准输出。
- 支持文本 token 比较。
- 支持浮点误差比较。
- 聚合 AC、WA、TLE、MLE、RE、CE 等状态。

适用场景：

- 普通 ACM 风格题目。
- 输出唯一或可通过标准输出判断的题目。

### 6.2 SpecialJudgeStrategy

用于 Special Judge 或构造题评测。

职责：

- 编译用户代码。
- 编译 Checker。
- 运行用户程序生成输出。
- 将输入、用户输出交给 Checker 判断。
- 解析 Checker 返回结果。
- 聚合最终评测状态。

适用场景：

- 构造题。
- 答案不唯一题。
- 需要自定义判断逻辑的题目。

### 6.3 OiJudgeStrategy

用于 OI 赛制下的题目评测。

职责：

- 按测试点执行评测。
- 根据通过情况计算题目得分。
- 保存单次提交得分。
- 返回每个测试点状态。
- 支持按题目最高分更新排行榜。

适用场景：

- 部分分题目。
- 训练赛、选拔赛、阶段性测验。

### 6.4 IoiJudgeStrategy

用于 IOI 赛制下的测试点或测试点组计分。

职责：

- 按测试点组执行评测。
- 支持测试点组内依赖关系。
- 支持测试点组全部通过才给分。
- 汇总测试点组得分为题目得分。
- 保存详细测试点明细。

适用场景：

- 测试点精细计分。
- 需要展示每个子任务得分的比赛。

## 7. 数据模型扩展建议

后续为支持 OI 与 IOI 赛制，可能需要扩展以下数据：

### 7.1 competition

建议增加或复用字段：

- `contest_mode`：比赛赛制，例如 `ACM`、`OI`、`IOI`。
- `score_visible`：比赛期间是否展示分数。
- `rank_frozen`：是否启用封榜。

### 7.2 competition_problem

建议增加字段：

- `score`：题目总分。
- `score_policy`：计分策略，例如最高分、最后一次提交分。

### 7.3 problem

建议扩展字段或配置：

- `judge_mode`：题目评测模式，例如 `standard`、`special`、`oi`、`ioi`。
- `score_config`：测试点或测试点组分值配置。
- `float_tolerance`：浮点误差。
- `checker_file`：Checker 文件。

### 7.4 submission

建议增加字段：

- `score`：本次提交得分。
- `max_score`：用户或队伍在该题上的历史最高分。
- `contest_mode`：提交时所属比赛赛制快照。

### 7.5 submission_case_result

建议新增测试点结果表，用于保存 IOI/OI 明细：

- `submission_id`。
- `case_index`。
- `group_name`。
- `status`。
- `time_used`。
- `memory_used`。
- `score`。
- `message`。

## 8. 排行榜计算规划

### 8.1 ACM 排名

ACM 排名继续按当前规则：

- 优先比较解题数。
- 再比较罚时。
- 可保留首杀、尝试次数等展示字段。

### 8.2 OI 排名

OI 排名建议：

- 按总分降序排序。
- 总分相同可按最后得分更新时间、提交次数或用户编号排序。
- 每题分数取该题历史最高分。

### 8.3 IOI 排名

IOI 排名建议：

- 按总分降序排序。
- 每题分数由测试点组得分汇总得到。
- 每题取历史最高分或按比赛配置取最后一次有效提交分。
- 支持展示每道题的子任务得分明细。

## 9. 前端组件化、性能优化与 Next.js 迁移规划

### 9.1 当前问题

当前前端仍存在以下可优化点：

- 页面级代码中存在较多重复布局、表单、表格、弹窗、筛选条件和状态处理逻辑。
- 业务组件边界不够清晰，部分组件职责偏大。
- 通用交互逻辑没有充分抽象为 Hooks。
- API 请求、加载态、错误态、空状态等处理方式不够统一。
- 缺少系统性的性能优化策略。
- 当前 Vite SPA 对 SEO、首屏性能、路由级数据加载和服务端渲染支持有限。

### 9.2 组件化拆分目标

后续前端应从页面驱动开发逐步转向组件驱动开发。

组件化拆分目标：

- 降低页面代码重复度。
- 提高管理后台、比赛页面、题目页面、提交页面的复用能力。
- 统一表单、列表、弹窗、状态展示等交互体验。
- 让页面只负责编排业务，具体 UI 和交互下沉到组件与 Hooks。
- 为后续 Next.js 迁移降低页面复杂度。

### 9.3 推荐组件分层

建议按以下层次组织前端组件：

```text
frontend/src
    → components
        → ui
        → layout
        → form
        → table
        → feedback
        → business
    → hooks
    → services
    → stores
    → utils
    → types
```

各层职责建议：

- **ui**：按钮、输入框、选择器、卡片、标签、徽章等基础组件。
- **layout**：页面布局、管理后台布局、内容容器、导航、侧边栏。
- **form**：通用表单项、文件上传、Markdown 编辑、代码编辑器封装。
- **table**：分页表格、筛选栏、批量操作栏、空状态。
- **feedback**：加载态、错误态、确认弹窗、Toast、结果状态。
- **business**：题目选择器、比赛题目列表、提交状态标签、排行榜表格、防作弊事件表格等业务组件。
- **hooks**：分页查询、权限判断、WebSocket 订阅、防作弊采集、表单提交等逻辑复用。
- **services**：按领域封装 API 调用，例如 `problemService`、`competitionService`、`submissionService`。

### 9.4 优先拆分方向

优先从重复最多、变化最频繁的页面开始拆分：

- **管理后台表格页**：用户、题目、比赛、练习、权限、AI 模型等页面可共用筛选、表格、分页和操作栏。
- **题目相关页面**：题目卡片、难度标签、标签列表、提交入口、样例展示、Markdown 渲染。
- **比赛相关页面**：比赛状态、报名状态、题目列表、排行榜、团队信息、防作弊风险展示。
- **提交相关页面**：提交状态、语言标签、耗时内存展示、队列位置、评测详情。
- **通用弹窗与表单**：创建、编辑、删除确认、导入、导出、上传等交互。

### 9.5 前端性能优化方向

后续应补充系统性的性能优化能力：

- **路由懒加载**：管理后台、AI、比赛详情、题目详情等大页面按需加载。
- **组件懒加载**：Monaco Editor、Markdown 编辑器、图表等重组件延迟加载。
- **资源拆包**：将编辑器、图表、Markdown 渲染等依赖拆成独立 chunk。
- **请求缓存**：题目标签、权限码、用户信息、比赛基础信息等低频数据增加缓存。
- **请求去重**：同一页面重复请求同一资源时复用请求结果。
- **列表虚拟滚动**：长提交列表、消息列表、防作弊事件列表按需启用虚拟滚动。
- **渲染优化**：合理使用 `memo`、`useMemo`、`useCallback`，减少大表格和复杂页面的无效重渲染。
- **WebSocket 优化**：统一连接管理、重连策略、订阅清理和消息分发。
- **图片与静态资源优化**：头像、题面图片、公告图片等资源按需加载。
- **构建分析**：引入 bundle 分析，持续关注首屏包体积。

### 9.6 Next.js 迁移目标

后续可评估从当前 React + Vite SPA 迁移到 Next.js。

迁移目标：

- 改善首页、题目列表、题目详情、比赛列表等公开页面的 SEO。
- 提升首屏加载体验。
- 使用文件路由统一页面组织。
- 支持服务端渲染或静态生成。
- 为后续内容型页面、题解、公告、比赛介绍等提供更好的索引能力。
- 保留需要强交互的在线编辑、评测、AI 对话、管理后台等客户端能力。

### 9.7 Next.js 迁移策略

建议采用渐进迁移，而不是一次性重写。

推荐步骤：

1. **组件先行**：先在当前 Vite 项目中完成基础组件、业务组件、Hooks 和 services 拆分。
2. **路由梳理**：整理当前页面路由，区分公开页面、登录用户页面、管理后台页面和强交互页面。
3. **API 层隔离**：将 axios 调用从页面中剥离到 services，减少迁移时页面改动。
4. **状态管理收敛**：梳理全局状态，避免页面依赖隐式全局副作用。
5. **建立 Next.js 原型**：先迁移首页、题目列表、题目详情等公开页面。
6. **迁移强交互页面**：在线编辑、提交详情、AI 对话等页面使用 Client Component 保留浏览器能力。
7. **迁移管理后台**：管理后台可最后迁移，也可以继续保持 CSR 模式。
8. **统一部署链路**：调整 Nginx、后端 API 代理、WebSocket 代理和静态资源部署方式。

### 9.8 Next.js 迁移注意事项

- **WebSocket**：评测结果、AI 流式、通知和私信仍依赖浏览器端连接，需要放在 Client Component 中。
- **Monaco Editor**：编辑器依赖浏览器环境，不能直接在服务端渲染。
- **权限判断**：需要区分服务端可判断权限和客户端动态权限。
- **Token 存储**：迁移后需要重新评估 Cookie、Header、SSR 鉴权和 CSRF 风险。
- **部署模式**：需要明确使用 Node Server、静态导出还是混合部署。
- **API 代理**：Next.js 与后端 `/api/v1` 路径需要避免冲突。
- **样式系统**：迁移时保持 TailwindCSS 和手写组件体系一致。
- **成本控制**：优先迁移能带来 SEO 或首屏收益的页面，管理后台不应作为第一阶段目标。

## 10. 与现有能力的关系

后续重构应尽量复用现有能力：

- Go-Judge 多端点 capacity 调度。
- RabbitMQ 异步评测队列。
- WebSocket 评测结果推送。
- `queueAhead` 队列位置展示。
- 标准题输出比较。
- Checker 特判能力。
- 浮点误差容忍能力。
- 比赛个人赛与团队赛能力。
- 现有 React + Vite 页面和 TailwindCSS 样式体系。
- 已沉淀的手写业务组件与 API 封装。

重构时应避免破坏现有 ACM 比赛与普通题目提交流程。

## 11. 推荐落地顺序

建议按以下顺序演进：

1. **抽象统一评测结果模型**：先定义总结果和测试点明细结果。
2. **拆分 StandardJudgeStrategy**：将当前标准评测逻辑迁移到独立策略类。
3. **拆分 SpecialJudgeStrategy**：将 Checker 相关逻辑迁移到独立策略类。
4. **增加策略选择器**：根据题目配置选择对应评测策略。
5. **引入 OI 基础计分**：先支持题目总分和按测试点累加得分。
6. **引入 IOI 测试点组**：再支持测试点组、子任务和更复杂的得分规则。
7. **重构排行榜**：根据比赛赛制选择不同排名计算器。
8. **完善管理端配置**：支持赛制、分值、测试点组、分数展示等配置。
9. **前端组件化治理**：优先拆分管理后台、题目、比赛、提交等高重复页面。
10. **前端性能优化**：补充懒加载、缓存、请求去重、渲染优化和构建分析。
11. **Next.js 迁移预研**：先迁移公开页面，再评估强交互页面和管理后台。

## 12. 风险与注意事项

- **兼容性**：已有 `standard` 和 `special` 题目必须保持兼容。
- **数据迁移**：新增字段需要提供默认值，避免影响历史数据。
- **排行榜性能**：OI/IOI 排名可能涉及更多得分明细，需要考虑缓存或预聚合。
- **测试点明细存储量**：测试点结果表可能增长较快，需要分页、归档或清理策略。
- **团队赛适配**：OI/IOI 排名需要同时支持个人和团队两个维度。
- **WebSocket 推送**：评测结果推送需要兼容状态型结果与分数型结果。
- **管理端复杂度**：测试点组配置应尽量降低操作门槛，避免管理员配置错误。
- **前端迁移成本**：Next.js 迁移不应与业务功能开发强耦合，应分阶段推进。
- **SSR 兼容性**：Monaco Editor、WebSocket、防作弊采集等浏览器能力需要明确客户端边界。
- **组件抽象过度**：组件化应优先解决重复和维护成本，不应为了抽象而抽象。

---

## 13. 总结

后续比赛系统将从单一 ACM 赛制扩展为 ACM、OI、IOI 多赛制并存。

评测系统也应从当前偏流程式、硬编码判断的实现，逐步演进为基于评测模式策略类的可扩展架构。

通过将标准评测、Special Judge、OI 计分、IOI 测试点组计分拆分为独立策略，可以降低主评测流程复杂度，提高系统后续维护和扩展能力。

前端侧也需要同步推进组件化拆分和性能优化，先解决当前页面代码冗余和无系统性性能治理的问题，再以渐进方式评估向 Next.js 迁移。
